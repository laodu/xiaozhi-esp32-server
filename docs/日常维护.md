# 日常维护操作

本文档记录本仓库（`xiaozhi-esp32-server`）在日常开发与部署中的**常用操作**，便于本人或协作者快速执行。**后续若有新流程、新环境约定，请在本文件中持续补充**（可在文末「变更与补充记录」追加条目）。

> 通用部署说明仍以官方文档为准：`docs/Deployment.md`、`docs/Deployment_all.md`、`docs/docker-build.md`。

---

## 1. Git 远程说明

维护两套远程，便于与上游同步、同时在自己的 GitHub fork 上保存与部署拉取代码。

| 远程名 | 含义 | 示例（请按你账号替换） |
|--------|------|------------------------|
| `origin` | 个人 fork，日常 **push / 服务器 git pull** 的目标 | `https://github.com/laodu/xiaozhi-esp32-server.git` |
| `upstream` | 上游官方仓库，仅 **fetch / merge**，一般不 push | `https://github.com/xinnan-tech/xiaozhi-esp32-server.git` |

**首次配置（若尚未配置）：**

```bash
git remote add upstream https://github.com/xinnan-tech/xiaozhi-esp32-server.git
# origin 应指向你的 fork，例如：
# git remote add origin https://github.com/<你的用户名>/xiaozhi-esp32-server.git
```

**查看当前远程：**

```bash
git remote -v
```

## 2. 日常同步上游（推荐流程）

目标：让本地 `main` 与上游一致，再推送到自己的 fork，保持 **本地 ≈ fork ≈ 已合并的上游最新**。

顺序如下：

1. **拉取上游最新引用**（不直接改工作区文件）  
   ```bash
   git fetch upstream
   ```

2. **将上游合并进当前分支**（在本地完成同步）  
   ```bash
   git checkout main
   git merge upstream/main
   ```  
   若偏好线性历史，可使用 `git rebase upstream/main`（注意与协作约定；有冲突时在本地解决）。

3. **推送到 GitHub 上的 fork**  
   ```bash
   git push origin main
   ```

**说明：**

- 若几乎不从 GitHub 网页改默认分支，一般**不必**先 `git pull origin`；若团队会在远端直接改 `main`，可先 `git pull origin main` 再与上游合并，减少冲突。
- 使用 **GitHub CLI** 可查看登录账号与仓库：`gh auth status`、`gh repo view`。

---

## 3. 用本地覆盖 fork（慎用）

当 fork 上历史杂乱、且**确认 fork 上无需保留任何独有提交**时，可用本地（已与上游对齐）强制覆盖远端 `main`：

```bash
git fetch upstream
# 确认本地 main 与 upstream/main 一致后再执行
git push --force-with-lease origin main
```

`--force-with-lease` 比 `--force` 更安全：若远端被他人更新会拒绝覆盖。执行后 **fork 上与旧 `main` 不同的提交将不再存在于 `main`**，需要时请事先另建分支备份。

## 4. 服务器部署更新（概要）

以下描述常见「代码更新后让服务生效」的思路；**具体路径以服务器为准**。

典型目录结构（示例）：

- 仓库部署路径：`/opt/xiaozhi-esp32-server/repo/`
- 全量自建镜像 Compose：`main/xiaozhi-server/docker-compose_all.deploy.yml`（已纳入本仓库，**Python + 管理端均从源码构建**，不依赖上游 `web_latest` / `server_latest` 镜像）

**代码更新（通用）：**

1. SSH 登录服务器，进入 **`main/xiaozhi-server`** 目录（与 compose 中 `context: ../..` 一致）。
2. `git pull`（远端一般为个人 fork）。
3. 重建并启动（示例，文件名按实际使用）：
   ```bash
   docker compose -f docker-compose_all.deploy.yml build
   docker compose -f docker-compose_all.deploy.yml up -d
   ```
   若只改了某一服务，可只 build 该服务名以节省时间，例如仅管理端：
   ```bash
   docker compose -f docker-compose_all.deploy.yml build xiaozhi-esp32-server-web
   docker compose -f docker-compose_all.deploy.yml up -d xiaozhi-esp32-server-web
   ```

**各组件说明：**

| 服务 | 构建来源 | 说明 |
|------|----------|------|
| `xiaozhi-esp32-server` | 仓库根目录 `Dockerfile-server` | Python 语音核心 |
| `xiaozhi-esp32-server-web` | 仓库根目录 `Dockerfile-web`（Vue `manager-web` + Maven `manager-api` + Nginx） | 管理端；镜像名 `xiaozhi-esp32-server-web:local` |
| DB / Redis | Compose 内 `image:` | 一般无需自构建 |

- 仅修改挂载目录下的配置（如 `./data`、环境变量）时，可只 **重启对应容器**，不必每次全量 `build`。
- **数据库密码、Redis 密码**等请在服务器上改为强密码，并同步修改 compose 中 `environment`（勿将真实密码提交到公开仓库）。

详细镜像构建与变量说明见：`docs/docker-build.md`、`docs/Deployment_all.md`。
